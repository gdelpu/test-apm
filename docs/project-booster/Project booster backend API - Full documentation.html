This is a stateless API for the DEP project booster, it acts as a resource server. A Gitlab server is used as the authentication server to validate the consumer identity and roles. The resource owners are all users of the Gitlab server.

1. Technical overview

The goal of this chapter is to explain important technical aspects that should be known by the consumer to fully use this API.

1.1. HTTP verbs

This RESTful API tries to adhere as closely as possible to standard HTTP and REST conventions in its use of HTTP verbs:

Verb Usage

GET

Used to retrieve a resource

POST

Used to create a new resource

PUT

Used to update an existing resource

PATCH

Used to partially update an existing resource

DELETE

Used to delete an existing resource

1.2. HTTP status codes

This RESTful API tries to adhere as closely as possible to standard HTTP and REST conventions in its use of HTTP status codes:

Status code Usage

200 OK

The request completed successfully

201 CREATED

A new resource has been created successfully. The resource’s URI is available from the response’s Location header

204 NO_CONTENT

An update (or deletion) to an existing resource has been applied successfully

400 BAD_REQUEST

The request was incorrect. The response body will include an error providing further information

404 NOT_FOUND

The requested resource did not exist

401 UNAUTHORIZED

The authorization bearer token has not been provided or is invalid, see Authentication or a required secret is missing to access an external resource server, see Required secret error

403 FORBIDDEN

You do not have the necessary role to access or modify this resource, see Authorization

429 TOO_MANY_REQUESTS

Too many requests have been sent to the endpoint for a given time frame, see Rate limit

500 INTERNAL_ERROR

An unexpected error occurred on the server

502 BAD_GATEWAY

A external server was contacted but the connection failed, either the server is not reachable or the server certificate is not trusted, see Untrusted certificate error
Some other specific client error code 4** can be sent back for technical errors such as 405 Method not allowed and 415 Unsupported media type.

1.4. Hypermedia

RESTful API uses hypermedia and resources include links to other resources in their responses. Responses are in Hypertext Application from resource to resource. Language (HAL) format. Links can be found beneath the links key. Users of the API should not create URIs themselves, instead they should use the above-described links to navigate.

An example is given below:

1.4.1. Response body

{
  "message" : "This is the index of the project booster api, check the html documentation in the link below",
  "links" : {
    "html-docs" : {
      "href" : "https://project-booster.dep.soprasteria.com/api/docs"
    }
  }
}

1.5. Authentication

This API requires a valid token to access to its resources. There are some exceptions for endpoints which are publicly visible and do not require it (access to this documentation is a good example).

Failing to provide a token when necessary will result in the response given below:

{
  "status" : "UNAUTHORIZED",
  "timestamp" : "2013-11-10T00:00:00.000000Z",
  "message" : "Full authentication is required to access this resource"
}

There are several ways you can authenticate to this API but all of them require you to send an Authorization header containing a Bearer token. An example is given below (where valid_private_token1 represents your token):

GET /api/users/current HTTP/1.1
Authorization: Bearer glpat-rx6x2AJr.USER1
Host: project-booster.apps.ocp4.innershift.sodigital.io:443

The following chapters present the different ways to get this token value. They all imply to generate a token through Gitlab, since the authentication is actually validated using the Gitlab server (acting as an authorization server for this API).

1.5.1. Gitlab private token

You can use a gitlab private token with api scope. This is the preferred authentication method to use this API. You can generate a new private access token on Gitlab in your user profile.

1.5.2. Gitlab Oauth2 token

You can use a gitlab oauth2 token with api+write_repository+openid scopes. This is the authentication method that is used implicitly when you are using this API from the UI front end application. This token is obtained through an Oauth2 workflow (in our case, authorization code with PKCE).

1.5.3. Gitlab OpenID token

You should not use this authentication method, it is only listed to be exhaustive.

You can use a gitlab openid token with api+write_repository+openid+email scopes. This is an alternative way to authenticate to this API, but it comes with a limitation because the JWT that is used for this method does not allow the API to query Gitlab on your behalf and thus limits you in the use of this API.

1.6. Authorization

Some endpoints of this API are protected based on the role of the consumer. When you do not have the appropriate role, you will face the below response:

{
  "status" : "UNAUTHORIZED",
  "timestamp" : "2013-11-10T00:00:00.000000Z",
  "message" : "Access Denied"
}

Endpoints that require a specific role will be indicated in this documentation.

2. Resources

This chapter presents the different resources accessible and manageable from this API.

This API has two main goals:

  • It supports the frontend application by providing data which are dynamically loaded from configuration files and represent the different user choices.

  • It allows to manage and execute scenarios which help a Sopra Steria project to get started on new technology, improve the industrialisation level of your project, embrace the DevSecOps approach and so on…

2.1. Orchestrators

An orchestrator represents a container orchestrator such as Kubernetes or Openshift.

This resource exists for the frontend application, it can be used to get information about orchestrators.

All the endpoints will require some secrets to work properly. Failing to provide the secret will generate a Required secret error.

Endpoints:

  • /api/certificates

  • /api/orchestrators/{type}

  • /api/orchestrators/{type}/namespaces

  • /api/orchestrators/{type}/namespaces/{namespace}/quotas

  • /api/orchestrators/{type}/namespaces/{namespace}/services

2.2. Artifactory

An artifactory represents an artifact repository such as Jfrog SSG Artifactory.

This resource exists for the frontend application, it can be used to get information about an artifactory.

All the endpoints will require some secrets to work properly. Failing to provide the secret will generate a Required secret error.

Endpoints:

  • /api/artifactory/repositories

2.3. Metrics

These resources allow to get some metrics about the application.

In this API, we observe the following metrics:

  • The number of unique daily connections to the API: Uniqueness is defined by the username of the authenticated user. At the end of each day, a CRON job adds a new metric entry with the number of unique connections and resets the table in which we store each unique users that connected that day.

  • Measure on the number of scenarios created, executed and pipeline actions performed on the scenarios.

Endpoints:

  • /api/metrics/connections

  • /api/metrics/usage

2.4. Config

The configuration endpoint is dedicated to the frontend application.

As its name suggests, it outputs a configuration that is loaded and used by the frontend app and which allows this application to be dynamic and easily modifiable.

The idea is to that we configure the app through a set of files stored in GIT repositories and we can reload these files in production when we update them. See the README of Project Booster to learn more about this (it is not required to use that API).

Endpoints:

  • /api/config

  • /api/config/authentication

2.5. Scenarios

Scenarios are the main resources of this API.

A scenario represents the goal you want to accomplish to get started on new technology, improve the industrialisation level of your project, embrace the DevSecOps approach and so on…​

By creating a scenario, you tell Project Booster what end result you expect. You can then get a preview of what you will get. And, finally, you can trigger (i.e. create) a scenario pipeline in order to let Project Booster automatically execute the actions required to fulfill that scenario.

The hierarchy of resources is described below:

Scenario   (1) → (n)   Pipeline   (1) → (n)   Action
                                      → (n)   Credential

(1) → (n) means for 1 scenario you can have n associated pipelines. Also it is true internally, you will only be able to get the latest pipeline from this API.

A scenario is a generic resource but it is typed. You must specify the type of scenario that you want when you create it. Each scenario type will require a different set of inputs.

Endpoints:

  • /api/scenarios

2.6. Pipelines

A pipeline represents the context in which one or several actions can be executed in order to complete a scenario. After creating a scenario, you can trigger (i.e. create, execute) a new pipeline for that scenario. The actions executed in the pipeline depend on the type of the scenario.

Pipelines run in the background. You can track their progress and cancel them asynchronously.

Endpoints:

  • /api/scenarios/{id}/pipeline

  • /api/scenarios/{id}/pipeline/credentials

3.5. /api/users/current/secrets

Manage secrets for the currently logged-in user.

The goal of this endpoint is to create secrets that can then be reused on server side by other endpoints.

Secrets can be created preemptively, or created when receiving a Required secret error.

Secrets are stored in memory and lost after a certain amount of time and can be all removed by calling DELETE /api/users/current.

Secret values can only be retrieved on server side (they are not accessible through this API) using their key, which uniquely identifies them for the current user.

Clients of this API can retrieve metadata about the secrets currently stored for them.

3.26.1. GET

Get the credentials created by the latest pipeline of a given scenario.

Expected status code(s) (does not include 401 Authentication error and 500 Generic error).

  • 200 OK with the pipeline credentials,

  • 404 NOT_FOUND if the scenario does not exist or the user is not the owner of that scenario (and the user is not an admin),

  • 404 NOT_FOUND if the pipeline for the scenario has not been triggered yet.
